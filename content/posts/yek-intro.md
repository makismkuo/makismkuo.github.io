## 仓库即上下文

如果你经常用 AI 写代码，一定遇到过这种情况：想把整个项目上下文喂给 ChatGPT 或 Claude，要么手动复制粘贴每个文件，要么写脚本 `cat` 拼接。赶上大项目，光是过滤 `.gitignore`、排序文件、控制 token 数就够折腾半天。

[yek](https://github.com/mohsen1/yek) 就是解决这个痛点的工具。一句话概括：**一个能把整个 Git 仓库序列化成单文件的 CLI 工具，专为 LLM 上下文设计。**

项目诞生于 2025 年初，目前 2400+ Star，作者是 GitHub 上知名的 Rust 开发者 mohsen1。名字 "yek" 在波斯语里是"一"的意思——把一堆文件变成"一个"文件，名副其实。

## 核心功能

yek 做的事情看起来简单，但细节做得相当到位：

- **自动忽略**：默认读取 `.gitignore`，同时内置了二进制文件、超大文件的过滤规则，不用手动配置
- **智能排序**：根据 Git 提交历史，把最常修改的文件排在输出的末尾。注意是末尾——因为 LLM 对越靠后的内容关注度越高
- **Token 感知**：`--tokens 128k` 可以直接按 token 数截断，超出的文件自动丢弃，保留最重要的部分
- **多目录和 glob 支持**：可以同时处理 `src/` `tests/`，也支持 `"src/**/*.ts"` 这样的通配符
- **YAML 配置**：项目根目录放一个 `yek.yaml`，可以自定义优先级规则、额外忽略模式、输出模板

## 为什么值得关注

同类工具不是没有，比如 Repomix 也做同样的事。但 yek 直接用性能把差距拉开了一个数量级。

**serialize 整个 Next.js 仓库**（超过 7000 个文件），yek 只需要 5 秒，而 Repomix 跑了 22 分钟。**230 倍的差距**，更不用说 yek 每秒序列化 108MB 的惊人吞吐量。

这个性能差距来自 Rust 的并行处理能力。yek 在多个层面做了并发：文件遍历、内容读取、token 计算——所有步骤都并行执行。

性能之外的另一个亮点是 Git 感知排序。大多数序列化工具只是按字母序排文件，yek 会分析 Git 历史，把核心业务代码排在更重要的位置。这种对 LLM 上下文窗口的精细化利用，体现了作者对实际使用场景的深入思考。

## 使用示例

安装非常简单，一行命令：

```bash
curl -fsSL https://azimi.me/yek.sh | bash
```

最常用的几个场景：

**1. 把整个仓库喂给 LLM**

```bash
yek
```
输出写入临时文件，路径直接打印在终端，复制粘贴就行。

**2. 只处理 src 目录，直接管道到粘贴板（macOS）**

```bash
yek src/ | pbcopy
```

**3. 限制 token 数**

```bash
yek --tokens 128k
```
超过 128K token 的文件会被自动截掉，优先保留核心代码。

**4. 自定义优先级**

在项目根目录创建 `yek.yaml`：

```yaml
priority_rules:
  - score: 100
    pattern: "^src/lib/"
  - score: 90
    pattern: "^src/"
```

这样 `src/lib/` 下的文件会排在最末尾（最优先被 LLM 关注）。

**5. 输出目录树**

```bash
yek -t src/
```
在文件内容前面加一个目录树，方便 AI 理解项目结构。
