你跟 AI 编码代理聊天时，是不是总是先拖一堆文件进 context？或者用 repomix 把整个仓库打包成一个 prompt？这个过程看着进度条慢慢走，还挺煎熬的。

**yek** 就是来终结这件事的。一个用 Rust 写的命令行工具，把任意目录的代码序列化成单一文件，方便喂给 LLM——而且比 repomix 快 230 倍。

## yek 是什么

yek（波斯语 "一" 的意思）是一个轻量 CLI，能把仓库里的文本文件按优先级排序后合并成一个大文件。默认行为就很聪明：它会读取 `.gitignore`，自动跳过二进制和大文件，还会用 git 提交历史推断哪些文件更重要——最后输出的顺序是"最重要的文件放最后"，因为 LLM 更关注上下文末尾的内容。

安装只要一行：

```bash
curl -fsSL https://azimi.me/yek.sh | bash
```

## 核心功能

- **极致的速度**：Rust + 并行处理。序列化整个 Next.js 仓库只要 5 秒，repomix 要 22 分钟。
- **智能排序**：基于 git 历史给文件打分，重要的放后面（LLM 注意力更集中）。
- **Token 感知**：支持 `--tokens 128k` 按 token 数截断，而不是盲目按字节截断。
- **配置灵活**：`yek.yaml`/`yek.toml`/`yek.json` 支持自定义优先级规则、忽略模式、输出模板。
- **流式输出**：检测到管道输出时自动流式传输，可以 `yek src/ | pbcopy` 直接复制到剪贴板。
- **多目录支持**：一次处理多个目录或文件，甚至 glob 模式。

日常用法非常直接：

```bash
# 默认输出到临时文件
yek

# 管到剪贴板
yek src/ | pbcopy

# 限制 token 数
yek --tokens 128k

# 加上目录树头
yek -t
```

## 为什么值得关注

用 AI 写代码的开发者越来越多，但大多数人还在手动选文件、复制粘贴。yek 解决的问题很实在：把你整个仓库的结构和代码一次性塞给 LLM，让 AI 拥有完整上下文。

它的竞争对手 repomix 也有类似功能，但差了两个数量级的性能——yek 用 Rust 重写了核心，串行变并行，把"等半分钟"变成了"瞬间完成"。对于每次向 AI 提问都要重新打包上下文的日常工作流来说，这个差距意味着你更愿意用它，而不是嫌慢而放弃。

还有一个小巧思：yek 会优先输出重要文件到末尾，利用 LLM 的"末尾偏重"特性。这种对模型行为模式的理解，让它比普通代码打包工具聪明不少。

## 总结

yek 解决的问题简单但高频：把代码仓库转化成 LLM 友好的格式。它的实现也毫不拖泥带水——Rust 写、并行处理、git 智能排序、token 感知截断。如果你经常跟 AI 编码代理配合工作，装上 yek 等于给自己的工作流加了个涡轮增压。
