## 写提示词前的准备工作

用 AI 写代码的时候，最烦人的环节是什么？对我来说不是写提示词本身，而是**把相关代码喂给 AI**。

项目规模小还好说，碰上几十个文件的大项目，手动复制粘贴显然不现实。用 repomix 之类的工具吧——老实说，跑到一半去泡杯咖啡回来还没跑完，也是常有的事。

yek 就是来解决这个问题的。

它是一个用 Rust 写的命令行工具，功能很简单：把一个代码仓库的所有文本文件串成一个文件，直接输出给 LLM 用。没什么花哨的 UI，没什么复杂的配置，装了就能跑。

## 核心功能

yek 做的事情非常明确，但细节做得挺到位：

**默认行为就很聪明。** 它会自动读取项目的 `.gitignore`，帮你过滤掉不需要塞给 LLM 的文件。node_modules、编译产物、二进制文件——这些东西默认就不会出现在输出里。

**Git 历史排序。** 它分析你的 Git 提交历史，把那些经常修改的核心文件排在输出的后面——因为 LLM 对上下文靠后的内容注意力更好。这个细节挺用心。

**Token 上限控制。** 你可以直接用 `--tokens 128k` 控制输出大小，超出上限的文件会自动被优先级低的文件替代。不会出现一刀切截断的情况。

**多目录和文件选择。** 不仅支持整个仓库，也支持指定目录、通配符模式、单个文件等多种输入方式。

## 为什么值得关注

yek 最主要的优势是**快**。有多快？

作者拿 Next.js 仓库做了个对比测试。yek 处理完整个仓库用了 5 秒，而 repomix 跑了 22 分钟——230 倍的差距。

不是"快一点"，是数量级上的碾压。

为什么快？因为 yek 是 Rust 写的，而且大量利用了并行处理。同样的任务，Node.js 版本的工具跑起来 CPU 在单线程上慢慢啃，Rust 版本直接把多核吃满。

另外它的输出格式也很合理：每个文件用 `>>>> file_path` 分隔，清晰明了，也可以自定义模板输出 JSON。默认直接写到临时文件并打印路径，也可以 `| pbcopy` 管道到剪贴板。

## 使用示例

安装非常直接：

```bash
curl -fsSL https://azimi.me/yek.sh | bash
```

最常用的场景，把整个项目串起来塞给 Claude/ChatGPT：

```bash
cd my-project
yek | pbcopy
```

这样就把当前目录下所有文本文件按重要性排序后复制到剪贴板了。然后直接粘贴到对话窗口就行。

只处理核心代码目录：

```bash
yek src/ | pbcopy
```

限制输出到 128K tokens（控制 LLM 上下文开销）：

```bash
yek --tokens 128k | pbcopy
```

每个目录单独输出到不同文件：

```bash
yek --output-dir /tmp/yek-output src/ tests/
```

如果想带上目录树结构方便 AI 理解项目组织：

```bash
yek -t | pbcopy
```

配置文件 `yek.yaml` 放在项目根目录，可以自定义优先级规则、忽略模式和输出格式。不过说实话，大部分时候默认配置就够用了。

## 和同类工具怎么选

目前市面上的类似工具主要是 repomix（Node.js）和 code2prompt（Node.js），还有早期的 grep + cat 手动方案。

我的建议很简单：

- **想要极致速度，项目体积大** → 直接上 yek
- **项目很小，顺手用** → repomix 也能满足需求
- **需要深度定制输出格式** → code2prompt 的模板系统更灵活

yek 还有一个隐藏优势：它是 Rust 写的二进制文件，安装后不依赖 Node.js 运行环境，在任何机器上都是一条命令的事。

## 总结

yek 不是一个改变世界的工具，它是一个把"把代码喂给 AI"这件事做到极致的工具。安装 5 秒钟，命令一行，结果是秒级的。

如果你经常和 AI 编程助手打交道，尤其是用 Claude、ChatGPT 或任何需要粘贴代码的对话式工具，yek 能省掉不少重复劳动。
