## 一个让人头疼的问题

维护一个项目久了，最怕的是什么？不是功能难写，不是 Bug 难修，而是代码越来越乱。

今天加一个模块，明天改一个接口，几个月下来，谁都说不清项目里各个文件之间的依赖关系到底是什么样的。某个底层模块被多少个上层文件引用？不知道。某个目录是不是已经变成了上帝类？不清楚。不同业务领域之间有没有偷偷耦合？猜不到。

代码规范、类型检查、linting 这些工具已经帮我们解决了很多层面的问题，但文件依赖和项目结构这块，一直缺少趁手的工具。

dep-tree 就是来填补这个空白的。

## dep-tree 是什么

dep-tree 是一个开源的 CLI 工具，用 Go 写成，核心能力是分析和可视化代码库的文件依赖关系。它支持 Python、JavaScript/TypeScript、Rust、Go 四种语言。

项目地址：[github.com/gabotechs/dep-tree](https://github.com/gabotechs/dep-tree)

目前 1700+ Stars，MIT 许可证，属于那种"知道的人不多但用过都说好"的工具。

## 核心功能

### 1. 3D 依赖图谱（Entropy）

这是 dep-tree 最酷的功能。输入一个入口文件，它会自动解析所有本地依赖，然后用浏览器渲染出一张 3D 力导向图。

每个文件是一个球体，球体之间的连线表示依赖关系。代码库的"熵值"会通过节点的分布直观呈现出来：

- **解耦良好的项目**：节点会自然聚集成若干簇，簇与簇之间有清晰的边界。这说明不同模块之间的职责划分明确，耦合度低。
- **耦合严重的项目**：所有节点糊成一团，没有清晰的聚类结构。这通常意味着项目在架构层面需要重构。

你可以像操作一个 3D 模型一样缩放、旋转、拖拽，从任意角度观察代码结构。这个可视化体验比看代码目录树直观太多了。

### 2. 文件依赖树（Tree）

如果你不习惯 3D 图，dep-tree 也提供了命令行的交互式树状图。用它来浏览一个文件的依赖链非常方便，支持键盘导航（上下翻、展开、回退），用户体验做得相当好。

### 3. 依赖规则检查（Check）

这个功能才是 dep-tree 的生产力所在。你可以通过 `.dep-tree.yml` 配置文件，定义哪些文件可以依赖哪些文件，哪些依赖是被禁止的。

举个例子：

```yaml
check:
  allow:
    "src/products/**":
      - "src/products/**"
      - "src/common/**"
  deny:
    "api/routes.py":
      - "adapters/**"
```

上面这个配置的意思是：`src/products` 目录只能依赖同目录和 `src/common` 里的文件，而 `api/routes.py` 不能引入 `adapters` 目录的任何内容。

把这个配置加到 CI 流程里，每次 PR 提交都会自动检查依赖是否越界。团队协作的时候，这个规则比口头约定靠谱一万倍。

### 4. 依赖关系解释（Explain）

当你想知道两个模块之间到底有什么依赖关系，用 `dep-tree explain` 就行了：

```bash
dep-tree explain 'src/products/**/*.go' 'src/orders/**/*.go'
```

它会列出所有从 products 模块指向 orders 模块的文件依赖。对于审查架构违规和重构前的依赖摸底，这个功能非常实用。

## 为什么值得关注

**第一，它解决了一个真实存在的痛点。**
每个中大型项目最后都会面临依赖管理的问题。市面上有 type checker（类型检查）、linter（代码规范）、formatter（格式化），却没有一个专门管文件结构和依赖的工具。dep-tree 填补了这个生态位。

**第二，可视化让人一目了然。**
一张 3D 图胜过千言万语。把 dep-tree 生成的图谱贴在团队 Wiki 或 PR 描述里，比写长篇大论的架构文档有效得多。

**第三，CI 集成让架构规则可执行。**
很多团队的架构规范只在入职培训时讲过一遍，之后就没人记得了。dep-tree 的 check 功能把抽象约定变成了机器可执行的规则，这才是工程化落地的方式。

**第四，跨语言支持。**
一个工具覆盖 Python、TS/JS、Rust、Go，对于使用多种语言的团队来说，降低了不少学习成本。

## 快速上手

安装 dep-tree 很简单，三种方式任选：

```bash
# macOS / Linux
brew install dep-tree

# 或者用 pip
pip install python-dep-tree

# 或者用 npm
npm install @dep-tree/cli
```

装完之后，找一个项目试试入门的两步操作：

```bash
# 1. 生成依赖树的 3D 可视化图谱
dep-tree entropy src/index.ts

# 2. 检查依赖规则（需要先创建 .dep-tree.yml）
dep-tree check
```

浏览器会自动打开，你就能看到项目的依赖结构了。

## 不足与局限

客观说几句 dep-tree 目前还不太完善的地方：

- **仅支持四种语言**。C++、Java、Ruby 等项目暂时用不了。
- **只分析本地文件依赖**，不追踪外部库的引入关系。这其实是一个合理的设计决策——外部库已经是既成事实，真正需要管理的是内部模块的边界。
- **Rust 支持标注为 beta**，遇到一些复杂的宏展开场景可能不够准确。

但这些并不影响它作为一个日常使用工具的实用性。对于 TS/JS 和 Python 项目，dep-tree 已经做得相当成熟了。

## 总结

dep-tree 解决的是代码架构中一个常被忽视的问题——文件级依赖管理。它用 3D 可视化的方式降低了理解代码结构的门槛，又用 CI 集成的 check 机制让架构规范从"纸上谈兵"变成了"机器执行"。

如果你正在维护一个逐步膨胀的项目，或者团队里经常因为"谁该引用谁"这种事争论不休，不妨装一个 dep-tree 看看。虽然它不能替你写代码，但至少能让你看清自己代码的真实结构——而这往往是重构的第一步。
