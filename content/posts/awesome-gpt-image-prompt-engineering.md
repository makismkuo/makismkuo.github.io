前段时间在研究 AI 图像生成，我之前写过一篇 LoRA 训练的文章，讲的是用 Flux 微调模型来稳定输出角色形象。那是从"模型侧"解决问题——让模型学会画特定的人。但这周我深入看了一个完全相反的思路：不碰模型，纯靠提示词工程，把 GPT-Image2 的输出精度推到接近工业级水准。

这个项目叫 **awesome-gpt-image-2**，在 GitHub 上维护，目前已经收录了 **476 个逆向工程案例**和 **20+ 套工业级提示词模板**。一开始我以为这就是又一个"提示词大全"——网上那种"50个超好用的AI绘画提示词"之类的东西。翻进去才发现，它的野心比那大得多。

## 这个项目在做什么

项目的核心主张很简单：**把散文式的提示词压缩成可复用的结构化协议。**

GPT-Image2 出来后，AI 图像生成行业的核心问题已经从"能不能生成图片"变成了"能不能稳定地、可控地、可复用地产出图片"。社区里散落着大量优秀的单次案例，但没有人系统地去拆解它们——哪些词起了作用？什么样的结构能稳定输出？哪些描述其实是废话？

项目作者做的就是把这些案例一个个扒开来看，提取出可复用的模板。476 个案例，每一个都是反向工程过的。这不是"我收集了 476 张好看图片"，而是"我拆了 476 次提示词，找到了它们的共同模式"。

## 13 个分类：覆盖了几乎所有我能想到的场景

项目把 476 个案例分成了 13 个类别。我列一下，你能感受到这个覆盖面：

1. **UI & 界面**（73 例）— 应用截图、网页设计、仪表盘
2. **图表与信息可视化**（51 例）— 流程图、对比图、缩放信息图
3. **海报与排版**（74 例）— 活动海报、运动 Campaign、概念字体海报
4. **商品与电商**（37 例）— 产品主图、包装、详情页
5. **品牌与 Logo**（23 例）— Logo 设计、品牌触点
6. **建筑与空间**（11 例）— 建筑渲染、室内设计、城市地图
7. **摄影与写实**（62 例）— 人像、街拍、商业摄影
8. **插画与艺术**（49 例）— 不同画风、材料实验
9. **角色与人像**（22 例）— 角色设计、姿势参考
10. **场景与叙事**（17 例）— 故事板、世界构建
11. **历史与国风**（16 例）— 古风、卷轴、传统题材
12. **文档与出版**（10 例）— 白皮书、手册、版式设计
13. **其他**（28 例）— 创意实验、混合工作流

每个类别都有对应的模板页，连"直播带货界面"和"水墨双重曝光海报"这种非常垂直的场景都有专门的模板。比如 UI 类里甚至细分了 iOS/Android/Web 截图模板、直播界面模板、截屏社交内容模板——每一种都针对不同平台的特征做了约束。

## 让我最惊讶的部分：模板的颗粒度

我原本以为这类"提示词模板"项目能做到"给你一个填空模板"就算不错了。但这个项目远远超过了这个水平——它给每个场景都准备了**三层工具**。

### 第一层：填空模板

最基础的，开箱即用：

```text
为[产品类型]生成一张[平台]界面图。
核心功能：[功能点A]、[功能点B]。
视觉风格：[极简/科技/拟物]，主色[颜色]，强调色[颜色]。
布局：[顶部导航/双栏/卡片流]，信息层级清晰，留白充足。
输出：高保真UI截图，文字清晰可读，比例[9:16/16:9]。
```

说实话，光有这个就已经比大多数网上看到的提示词示例强了——它把布局、风格、输出格式全拆成了独立维度，而不是混在一段散文里。

### 第二层：JSON 进阶模板

但真正有意思的是这个——专门为 AI Agent 调用设计的 JSON 格式：

```json
{
  "type": "Infographic",
  "topic": "Urban Metabolism",
  "audience": "General Public",
  "structure": {
    "title_area": "城市生命系统图谱",
    "layout": "Isometric cutaway, 12 numbered panels",
    "modules": [
      {"title": "能源", "icon": "lightning", "text": "Power flows"},
      {"title": "水循环", "icon": "water_drop", "text": "Water flows"}
    ]
  },
  "style": {
    "aesthetic": "Scientific atlas",
    "colors": "Low saturation, color-coded flows",
    "background": "Light paper texture"
  },
  "constraints": "No cyberpunk, no gibberish text, strict structural layout"
}
```

这意味着如果你在写一个自动化生成信息图的 Agent，可以直接用 JSON 结构来驱动提示词，而不是靠字符串拼接。模板的每个字段都对应图像生成的某个控制维度——这种"Prompt as Code"的思路，才是工程化的正确姿势。

### 第三层：避坑指南

每个模板后面都跟着一个"避坑指南"板块。这不是那种无关痛痒的"温馨提示"，而是真正从实战中摔出来的经验。比如 UI 模板的避坑指南：

- **不要给模糊指令**：明确"平台 + 比例 + 布局"，否则模型会像个实习生一样乱排版
- **强制文字锁定**：要求"文字绝对可读，必须显示指定的中文"，避免出现乱码
- **截图区分平台特征**：X（Twitter）有蓝勾认证和转发/引用的区分；抖音有音乐碟片和点赞动画；小红书有双列瀑布流特征。生成前必须指定平台，否则模型会混搭
- **直播界面先定场景**：带货直播和才艺直播的 UI 布局差异很大，带货右上角有商品列表，才艺直播重在弹幕互动

看到"模型会像个实习生一样乱排版"这句话的时候笑了一下——确实太真实了。用过 AI 生成 UI 的人应该都经历过那种"我要一个简洁的登录页，它给我塞了八个按钮和三个弹窗"的尴尬。

还有**海报模板的避坑指南**更狠："不要偷懒，写清主视觉到底是什么玩意儿，不要只丢一句'做一张海报'就指望出神图"。这种表达方式让我很舒服——它不是学院派的"最佳实践"，而是踩过坑的人用大白话告诉你别犯同样的错误。

## Agent Skill：把模板变成工具

项目还配套做了一个 npm 包（`gpt-image-2-style-library`），可以直接安装到 Claude Code、Codex 或 Cursor 里当 Agent Skill 用。安装后，AI 编程助手就获得了"GPT-Image2 风格库"的能力——你说"生成一张关于 Codex 的信息图"，它就能自动选择合适模板、填充参数、输出完整的结构化提示词。

安装方式也很简单：

```bash
npx skills add freestylefly/awesome-gpt-image-2 --skill gpt-image-2-style-library --agent claude-code codex
```

或者如果你在用 Claude Code，还可以走插件市场：

```
/plugin marketplace add freestylefly/awesome-gpt-image-2
/plugin install gpt-image-2-style-library@awesome-gpt-image-2
```

这个设计思路非常有前瞻性——它不再是把提示词写在文档里等人来读，而是把提示词变成 Agent 可以调用的工具。数据和 Agent 共享同一个 `style-library.json`，网站前端和 Agent 工作流共用一套风格库。一次维护，到处生效。

项目还配套搭了一套完整的 Web 应用——基于 Vite + Supabase + Vercel，有 Google 登录、Stripe 付费、案例收藏、GPT-Image2 代理生成。说实话这个完整度已经超出了"GitHub 项目"的范畴了，更像一个产品。

## 和 LoRA 训练的对比

我之前写的 LoRA 训练文章讲的是从模型侧保证一致性——你训练一个 LoRA，模型就学会了某个角色的脸型、服装、画风。好处是生成时几乎不需要操心提示词，坏处是训练成本高、迭代慢、换风格得重新训。

而这个项目走的是纯提示词工程路线。它的思路是：与其让模型"记住"你的需求，不如把提示词写得足够精确、足够结构化，让模型一次就理解你要什么。模板中的每一条"约束"、每一个"避免"项，都是在和模型反复交手后总结出来的有效指令。

两种思路各有适用的场景。LoRA 适合需要高度一致性的场景——比如一个角色的连续插图、一个品牌形象的统一输出。结构化提示词适合快速原型和多样化输出——你可以在几分钟内尝试完全不同的视觉方向，而不需要等模型重新训练。

但有意思的是，当我把这个项目的 JSON 模板和 Agent Skill 结合起来看时，我发现这两条路线可能在某个点会交汇。假设你的 Agent 能自动识别需求、选择合适的模板、填充业务参数、调用 GPT-Image2 生成、再根据结果迭代模板——那模型侧和提示侧的分界线会变得越来越模糊。最终，"你在用提示词还是用模型"这个问题本身可能就不重要了——重要的是你建立了一个能稳定产出高质量结果的系统。

## 为什么这个项目让我想写一篇文章

作为技术人，我习惯看到一个问题就想到"造个工具解决它"——LoRA 不行就换 ControlNet，ControlNet 不行就写个后端统一调度。但这个项目给我的启发是：有时候解决方案不在更复杂的工程里，而在对现有工具的更深入理解中。

项目作者花了大量时间去拆解、归类、验证那些有效的提示词模式。这种"逆向工程"的工作方式，比重复造轮子更接近技术的本质。它不需要你写多少代码，但需要你足够较真——476 个案例，每一个都拆开看，找到规律，抽象成模板，再反复验证。这种节奏跟我自己做各种项目测试时的感受很像：前期慢，但一旦找到模式，后面的效率会指数级上升。

我在部署 Telegram Bot、Crawl4AI、MediaCrawler 这些项目时也有类似的体验——真正花时间的地方从来不是写代码，而是理解"这个工具到底能做什么、不能做什么、在什么条件下表现最好"。这个项目把同样的思路用在了 AI 图像生成上：与其造一个新的图像生成工具，不如把现有工具的能力边界摸清楚，然后围绕它建立一套可靠的工程流程。
