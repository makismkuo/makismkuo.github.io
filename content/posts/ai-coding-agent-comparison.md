最近一个月，AI 编程 Agent 的赛道卷出了新高度。

先是 Anthropic 把 Claude Code 从实验性功能升格为独立 CLI 工具，代码库开源，star 数一周破万。紧接着 OpenAI 发布了 Codex CLI 的桌面版，带 GUI 界面、支持多窗口操作。Cline 也没闲着，更新了 MCP 支持和 OpenRouter 集成，让 VS Code 里的 AI 助手直接能读数据库、操作文件系统。

三家在同一个月内密集出牌，互相追赶的味道很浓。

但说句实话，工具越多越不知道该用哪个。每个都说自己"最懂代码"，但真正上手跑一个项目，体验天差地别。

我花了几天时间，用同一个项目（一个个人博客的 RSS 聚合服务）分别跑了这三款工具，从安装、使用体验、代码质量、成本到坑点，做了完整对比。

**结论先放这：没有绝对的"最好"，但各自的适用场景非常清晰。**

---

## 它们分别是什么

一句话定义：

**Claude Code** — Anthropic 推出的终端原生 AI 编程 Agent。直接在你的命令行里工作，能读文件、写代码、执行命令、看懂终端输出。

**Codex CLI** — OpenAI 的编程 Agent，最新版本有了桌面 GUI。既有终端模式也有可视化界面，核心是 GPT-4o/4.1 驱动的代码理解和生成能力。

**Cline** — VS Code 插件形态的 AI 编程助手。不是终端工具，而是在编辑器侧边栏开一个对话窗，能操作你的工作区文件、终端、浏览器。

如果你用过 GitHub Copilot，可以这样理解：

- Copilot 是"补全你正在敲的代码"——被动响应
- Claude Code / Codex CLI 是"你告诉它做什么，它从头做到尾"——主动执行
- Cline 介于两者之间——在编辑器里协作，你看着它干活，随时打断修正

## 快速上手：三款工具的安装对比

先看安装门槛。懒人可以直接看小结。

### Claude Code

```bash
# 方式一：npm 全局安装
npm install -g @anthropic-ai/claude-code

# 方式二：直接用 npx
npx @anthropic-ai/claude-code

# 进入项目目录
cd my-project
claude
```

安装后第一次运行会要求登录 Anthropic 账号并绑定 API Key。之后在项目目录下执行 `claude` 就会进入交互模式。

项目里会生成一个 `.claude` 目录，存会话记录和项目配置。

**最低要求**：Node.js 18+，有 Anthropic API Key 或 Claude Pro 订阅。

### Codex CLI

```bash
# 终端版
npm install -g @openai/codex

# 桌面版（推荐）
# 从 openai.com/codex 下载桌面应用
```

桌面版安装后启动的是一个本地 Web 界面（localhost:1080），能看到文件树、终端输出、对话历史。你可以给 Codex 分配任务，它在后台执行，你可以同时做别的事。

终端版更加轻量，用法跟 Claude Code 类似：

```bash
codex "帮我写一个 RSS 解析器"
```

**最低要求**：OpenAI API Key，Node.js 18+。

### Cline

Cline 不走 CLI 路线，是 VS Code 扩展：

1. VS Code 扩展市场搜 "Cline"
2. 安装后在侧边栏看到 Cline 图标
3. 首次使用配置 API Provider（OpenAI / Anthropic / OpenRouter / 本地 Ollama）
4. 设置模型、成本上限、允许 Cline 执行的操作（读文件、写文件、终端命令、浏览器）

Cline 可以在设置里打开"自动批准"模式，也可以每次操作都弹窗确认。

**最低要求**：VS Code，一个可用的 API Key。

### 💡 小结

| 工具 | 安装时间 | 依赖 | 学习成本 |
|------|---------|------|---------|
| Claude Code | 1 分钟 | Node 18+ | 低 — 会用终端就行 |
| Codex CLI | 2 分钟 | Node 18+ | 中 — 桌面版需要适应界面 |
| Cline | 30 秒 | VS Code | 低 — 就是装个插件 |

安装阶段 Cline 胜出——VS Code 装插件是每个前端的基本操作。但如果你的工作流不在编辑器中（比如你在服务器上 SSH 开发），Claude Code 和 Codex CLI 的终端模式是唯一选择。

## 实测：用同一个任务跑三遍

为了公平对比，我写了一个统一的任务描述，让三个工具分别完成：

> "在项目目录下创建一个 RSS 聚合服务。用 Node.js + TypeScript，支持同时抓取多个 RSS 源，去重后输出为 JSON 文件。要求：使用 axios 抓取，fast-xml-parser 解析，提供去重逻辑。"

### 1. Claude Code

Claude Code 接任务的方式是在终端里直接说：

```
claude
> 帮我创建一个 RSS 聚合服务，用 Node.js + TypeScript...
```

它做的第一件事：**读项目结构**。Claude Code 会自动扫描当前目录，理解已有的 `package.json`、`tsconfig.json` 等配置文件。然后它问了我几个问题：

- "输出文件路径有要求吗？"
- "去重是按 title 还是链接？"
- "要不要支持定时抓取？"

问完之后开始干活：创建文件、装依赖、写代码，一气呵成。

整个过程在我面前实时展开——它每写一个文件、每执行一条命令，终端都实时输出。如果报错了，它会自动读错误信息、修正、重试。

我第一次跑的时候，不知道是 `fast-xml-parser` 的导入语法有问题还是版本问题，它遇到一个 import 错误，自动执行了 `npm ls fast-xml-parser` 检查版本，然后改成正确的导入方式，继续运行直到成功。

**总耗时：从接任务到跑通，大约 4 分钟。**

### 2. Codex CLI

Codex CLI 桌面版的使用体验完全不同。

打开桌面应用后，先在左侧栏设置项目目录路径，然后在对话窗里输入同样的任务描述。

Codex 的处理方式偏"规划先行"——它先写一个详细的执行计划，包括文件结构、模块划分、依赖项列表，然后把计划展示给你看，等你确认后才开始动手。

这个模式的好处是可控。如果你对代码结构有特定要求，可以在执行前调整计划。坏处是多了一步确认流程，节奏慢一些。

实际执行时，Codex 的桌面版可以多窗口并行。它有多个 Agent 实例在跑——一个在写主文件，一个在装依赖，一个在写测试。我观察到的输出显示它同时开了 3 个子进程。

**代码质量**：Codex 生成的代码注释最多，类型定义最完整。另一个特点是它会在关键逻辑处留 error handling 的标记。

**总耗时：约 5 分钟（含确认计划的时间）。**

### 3. Cline

Cline 在 VS Code 侧边栏里工作。输入任务后，它先展示"计划做什么"，然后逐步骤执行。

Cline 的执行风格最"啰嗦"——每个步骤都会展示它要读的文件、要改的内容、要执行的命令，并等待你确认（如果没开自动批准）。

实际写代码的过程：Cline 创建文件 → 在编辑器中打开 → 展示 diff → 确认 → 下一个文件。这种节奏对于想"看着它干活"的人来说是安全感，但如果你只是想快点得到结果，会有点烦。

好在 Cline 支持"自动批准终端命令"和"自动批准文件写入"的独立开关，可以把文件写入设为自动，只对终端命令要求确认。

**总耗时：约 6 分钟（含人工确认的时间）。但如果把批准全开，大概 3-4 分钟。**

### 💡 代码质量对比

我用相同的测试数据（3 个 RSS 源）分别测试了三份输出：

| 维度 | Claude Code | Codex CLI | Cline |
|------|-------------|-----------|-------|
| TypeScript 类型覆盖 | 中等 | 完整 | 完整 |
| 错误处理 | 基础 try-catch | 分层错误处理 | 中等 |
| 性能考虑 | 有（并发控制） | 有（并发控制+超时） | 基础 |
| 注释密度 | 低 | 高 | 中 |
| 首次通过率 | 高（自修正能力强） | 高（计划阶段已排错） | 中（依赖人工确认） |

## 成本和限制

这是很多人容易忽略的部分——**用编程 Agent 写代码是实打实烧 token 的**。

### Token 消耗实测

同一个任务，三个工具的 token 消耗：

| 工具 | 模型 | 输入 tokens | 输出 tokens | 预估成本 |
|------|------|-------------|-------------|---------|
| Claude Code | Claude Sonnet 4 | ~85K | ~8K | $0.15-0.25 |
| Codex CLI | GPT-4o | ~120K | ~10K | $0.30-0.50 |
| Cline | 取决于配置 | ~70K | ~7K | 可变 |

Claude Code 的输入 token 相对少，因为它默认只读取相关文件（基于项目结构分析），不像 Codex 那样大量输出规划文本。

Cline 的成本完全取决于你绑哪个 API——如果绑本地 Ollama 跑 7B 模型，成本为零但代码质量下降显著。如果绑 Claude API，成本跟 Claude Code 接近。

**但有一个隐藏成本：重试。**

Claude Code 因为会自动修正错误，一次失败后重试产生的额外 token 会多 20-40%。Codex 的规划前置模式反而减少了执行阶段的失败率。Cline 的重试则取决于你的确认节奏。

### 硬约束

**Claude Code 的几个限制：**
- 重度依赖网络——没有离线模式
- 在超大项目（10万文件级别）中上下文窗口容易撑爆，需要手动配置 `.claudeignore`
- 终端独占——不擅长非代码任务（比如设计 UI 界面）

**Codex CLI 的几个限制：**
- 桌面版目前只支持 macOS（终端版无限制）
- 多 Agent 并行看起来很酷，但如果任务之间的依赖关系复杂，反而会互相干扰
- token 消耗偏高——规划文本占了很大一部分

**Cline 的几个限制：**
- 高度依赖 VS Code——换个编辑器就得换工具
- "每一步确认"的模式在复杂任务中容易打断思路
- 对超大文件的处理能力不如终端工具

## 什么场景该用哪个

做了这么多测试，其实结论挺清晰的：

### 用 Claude Code 的场景
- **你习惯在终端工作** — SSH 到服务器、tmux 会话、Neovim 用户
- **你写 Python / Go / Rust** — 这类语言的项目结构在终端里最自然
- **你希望 AI 自己 debug** — Claude Code 的自修正能力是三个里最强的
- **你不在意代码有没有注释** — 它的代码偏简洁，注释少但逻辑清楚

### 用 Codex CLI 的场景
- **你喜欢先看到计划再执行** — 需要掌控感的时候，Codex 的模式最舒服
- **你在写复杂的企业级项目** — 完整的类型定义、分层错误处理、详尽的注释
- **你有 OpenAI 的 API 额度** — Codex 的 token 消耗偏高，但代码质量在复杂场景下对得起这个成本
- **你想看着多个 Agent 同时干活** — 多窗口并行的视觉反馈很爽

### 用 Cline 的场景
- **你日常在 VS Code 里写代码** — 不需要离开编辑器
- **你希望用本地模型** — Ollama + Cline 的组合可以完全离线
- **你在学习/教学** — 逐行确认的模式天然适合理解 AI 生成的代码
- **你需要精细控制** — 每个步骤都能人工干预

### 不该用的时候

- **不要用编程 Agent 做架构决策** — 它们擅长实现，不擅长做权衡。项目架构应该自己想清楚
- **不要在生产环境直接跑生成的代码** — 无论哪个工具，代码审查都不能省
- **不要用它写你不理解的代码** — 复制粘贴看不懂的代码，调试的时候会哭

## 聊聊我的感受

跑了几次之后，我的感觉是：**这三个工具不是竞争对手，而是面向不同工作流的三种交互模式。**

Claude Code 代表的是"AI 作为协作者"——你在终端里指挥它，它自己干，你只在关键节点介入。适合那些已经熟练、知道自己要什么的人。

Codex CLI 代表的是"AI 作为规划者"——它先想后做，每一步都有据可查。适合那些需要确认感和文档完备性的场景。

Cline 代表的是"AI 作为工具"——装个插件就能用，不改变你的工作习惯。适合把 AI 当成开发流程中的一个插件而不是核心。

我自己的使用习惯是：**日常开发用 Cline**（因为 VS Code 是我的主编辑器），**复杂任务甩给 Claude Code**（自修正能力强，不用我盯着），**写正式项目用 Codex CLI**（代码质量高，还省了写注释的时间）。

三个工具加起来，每月 API 花费大约 20-30 美元。跟雇一个实习生比，便宜太多。

不过有一点我得诚实地说——这三个工具在大型项目（5万行以上）的表现都不太理想。上下文窗口再大，也装不下整个中大型项目的逻辑依赖。实际使用中，还是需要把任务拆解到模块级别再交给它们。

这可能是下一个要解决的问题：**AI 编程 Agent 的规模化能力**。但目前来看，5000 到 1 万行代码的项目，它们已经做得相当好了。

---

如果你也在纠结选哪个，建议按自己的编辑器偏好和对"控制感"的需求来选。Vim 用户 → Claude Code， 管理型 → Codex CLI，VS Code 用户 → Cline。
