## 凌晨三点，谁在替你写博客？

上周我在群里看到有人吐槽：自己搞了个 AI 博客助手，每天早上 8 点自动写一篇文章发布。结果某天打开网站，发现首页飘着一篇 **《论土豆的100种烹饪方法——某大型语言模型的厨房哲学思考》**。

底下一堆老外评论："Is this blog about tech or food?"

故事真假我不知道，但「让 AI 自己跑定时任务」这件事，确实正在从小众玩票变成正经需求。

我最近也在搞类似的事——用 Hermes Agent 的 Cron Job 功能，让 AI 每天按时起床，选主题、写文章、发博客，全程无人值守。跑了几周之后，踩了不少坑，也摸到了一些门道。

今天聊聊我实践下来的一套东西，不吹不踩，实打实记录。

---

## 它是什么

一句话：**AI Agent 定时任务，就是把 AI Agent 塞进 cron 里，按时间表自动执行复杂工作流。**

传统 cron 只能跑脚本——备份数据库、清理日志、发个 curl。脚本是确定性的，输入输出都在预期内。

AI Agent cron 不一样。它跑的是一个**有「判断力」的任务**——比如"今天选个有意思的开源项目写一篇博客"，或者"检查所有服务状态，有异常就处理"。它要自己决策、自己执行、自己收尾。

💡 **如果传统 cron 是闹钟，AI Agent cron 就是值班经理**——闹钟响了只能响，值班经理响了要判断要不要叫消防队。

---

## 为什么这事值得搞

### 痛点很明显

你不可能 24 小时盯着。Bot 挂了要重启、新内容要发布、数据要汇总——这些东西在人类不在场的时候也在发生。

以前这些事要么半夜爬起来弄，要么写死脚本（但脚本处理不了意外）。我试过用 Shell 脚本 + curl 做健康检查，一旦返回格式变了就废了，还得手动修。

### AI Agent 的优势恰好补上

- **能处理异常** — 不是固定 if-else，而是根据上下文判断
- **能写内容** — 不是模板填空，是真的"想"一篇出来
- **能跨工具** — 查 GitHub、读文件、写文章、git push，一整套流程串起来

听起来很美好对吧？实际上跑起来之后，才知道坑在哪。

---

## 架构长什么样

一个典型的 AI Agent 定时任务系统分三层：

```
┌─────────────────────────────────────────┐
│           调度层 (Cron/Scheduler)         │
│  每天10:00 / 每30分钟 / 周一早8点         │
├─────────────────────────────────────────┤
│           代理层 (AI Agent Runtime)       │
│  选模型 → 加载技能 → 执行工作流 → 输出     │
├─────────────────────────────────────────┤
│           执行层 (Tool Execution)         │
│  终端命令 / 文件读写 / API调用 / Git操作   │
└─────────────────────────────────────────┘
```

我用的方案是基于 **Hermes Agent**（Nous Research 的桌面端 Agent），它原生支持 Cron Job 调度。但这套思路放到 Claude Code、Codex CLI 或其他 Agent 框架上也大同小异。

## 具体配置

### 1. 定义任务

```yaml
# Hermes Cron 配置示意
schedule: "0 10 * * *"       # 每天早上10点
skills: ["blog-writing"]     # 加载的技能
enabled_toolsets: [terminal, file, web, search]  # 允许的工具
workdir: /path/to/blog
prompt: |
  今天的任务是：
  1. 检查当天是否有文章
  2. 选一个有趣的主题
  3. 写一篇2500-5000字的博客
  4. git add → commit → push
  5. 确认部署成功
```

几个关键参数：

- **schedule**：标准 cron 表达式。`0 10 * * *` 是每天 10:00，`*/30 * * * *` 是每 30 分钟
- **skills**：告诉 Agent 它有哪些"专业知识"可用。比如我写博客时加载写作规范技能，它就知道署名、格式、风格要求
- **enabled_toolsets**：限制能调用的工具。**非常重要**——你不想 Agent 半夜自己装了个 Homebrew 包
- **prompt**：任务指令。**越具体越好**，别指望 Agent 自己领会

### 2. 写一份好的 Prompt

这是最关键的环节。好的 cron prompt 长这样：

> 检查今天是否已有文章 → 如果有，跳过不写 → 如果没有，从以下方向选一个主题 → 完整生成文章 → 保存到指定路径 → 推送到远程 → 验证部署

坏的 prompt 长这样：

> 每天写一篇博客

后者大概率会产出《土豆的100种烹饪方法》。

💡 **原则：把 Agent 当新来的实习生。** 他聪明但不知道你的潜规则。每一步该怎么做、边界在哪、遇到什么情况要停，都写清楚。

### 3. 验证机制

任务跑完了，谁来确认它真的干好了？

我在任务末尾加了三步验证：

```bash
# 1. GitHub Pages 是否上线
curl -s -o /dev/null -w "%{http_code}" https://yourdomain.com/posts/today-post/

# 2. 反向代理是否同步
curl -s -o /dev/null -w "%{http_code}" https://mirror-site.com/posts/today-post/

# 3. RSS feed 是否包含新文章
curl -s https://yourdomain.com/rss | grep "文章标题"
```

如果某一步失败，Agent 会记录状态、尝试重试或跳过——而不是当没发生过。

## 坑点记录

跑了大概三周，翻车记录如下：

### 🔥 坑 1：Agent 在"自由发挥"和"严格执行"之间走钢丝

**现象**：某次 Agent 觉得文章不够好，自己决定重写一遍，然后把两篇都发布了。

**原因**：任务 prompt 里没强调"一篇即可"。Agent 的"追求完美"本能让它做了额外的事。

**解决**：prompt 里写死「如果当天已有文章，直接跳过。不要做额外的事。」

### 🔥 坑 2：Token 消耗比想象中大

**现象**：跑一次带搜索、读文件、写文章的完整流程，deepseek-v4 这种大模型一次吃掉 30K-50K tokens。

**原因**：Agent 在任务之间会过度确认——"我现在要开始写作了，确认一下"、"写作完成，要检查一下吗"——每一步都在消耗上下文。

**解决**：在 prompt 里加「普通问答不要检查环境，不要探测本机工具，不要搜索旧会话」。并且在工具配置里限制不必要的 self-check。

### 🔥 坑 3：失败重试的死循环

**现象**：网络不稳定，git push 失败。Agent 自动重试，又失败，再重试，连续 5 次后还在试。

**原因**：Agent 默认会为了完成任务不断尝试，这是它的优点也是缺点。

**解决**：在 prompt 里写「同类工具失败 2 次就停止尝试并说明原因」。设定明确的止损线。

### 🔥 坑 4：时区问题

**现象**：明明写的是凌晨 3 点跑，结果凌晨 3 点 UTC 也就是北京时间上午 11 点才动。

**原因**：cron 默认 UTC，没调时区。

**解决**：配置时显式写 `TZ=Asia/Shanghai`。

### 🔥 坑 5：Agent 写文章"机器人味"太重

**现象**：读起来像产品说明书，一段 500 字不带换行，大词小词堆在一起，看一眼就知道是 AI 写的。

**原因**：默认的 Agent 写作风格偏正式、详细。

**解决**：在技能里加入「说人话」、「像技术人平时说话的方式」、「短段 + 加粗 + emoji」等明确风格指令。并且我在技能里加了一句话：「如果用户大幅重写你的文章，尊重用户的版本。」——因为 AI 写的初稿确实经常需要人类润色。

## 哪些场景适合、哪些不适合

### ✅ 适合做的

- **定时内容生产** — 日报、周报、博客（有明确主题范围）
- **服务健康检查 + 自动修复** — 端口检测、进程重启
- **定时数据汇总** — 拉取 API 数据、生成报告
- **社交账号自动更新** — 推文、动态（但要注意内容审核）

### ❌ 不适合做的

- **涉及金钱交易的操作** — 付钱、转账、下订单（一次误操作代价太大）
- **需要人工审核才能发布的内容** — 除非你愿意承担翻车风险
- **对实时性要求极高的事** — Agent 跑一次流程可能几分钟到十几分钟
- **无明显边界的任务** — "研究一下AI的最新进展"这种范围太模糊

💡 **一个检查标准：这件事如果跑错了，最坏结果你能接受吗？** 如果不能接受，就别让 Agent 全自动。

## 聊聊我的感受

跑了几周的 AI 定时任务，最大的感受不是"AI 太强了"，而是「**边界感比能力更重要**」。

让 AI 做定时任务这件事，技术门槛其实不高——不就是把 API 调用扔进 cron 嘛。真正难的是：

1. **定义什么是"做得对"** — 对写脚本来说很容易（返回码 0），对 AI 来说很难（文章质量怎么量化？）
2. **设定怎么做不对** — 什么时候该停、什么情况下该放弃、什么错误可以忽略
3. **接受不完美** — AI 跑出来的东西不会每次都和你想的一样，有些 bug 你甚至不知道怎么复现

我现在的做法是：把 AI Agent 当成一个**很聪明但需要明确边界的新同事**。给它足够的上下文（加载技能），给它清晰的约束（prompt 里的规则），给它止损机制（失败 2 次就停），然后让它放手干。

至于它偶尔产出一篇风格奇怪的文章——只要不是我半夜爬起来处理的，我觉得都可以接受。
