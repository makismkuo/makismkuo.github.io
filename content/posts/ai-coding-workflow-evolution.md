2026 年春天，GitHub Copilot 宣布用户突破 5000 万。同一周，Claude Code 的仓库冲到了 20k+ star，Cursor 完成了新一轮融资，估值破百亿美元。

AI 编程工具已经不是新鲜事了。

但用过的人心里都有数——打开一个对话框让 AI 写代码，和真正用它做成一件事，中间隔着一条很宽的河。

**工具越来越多，但问题其实一直没变：该怎么跟 AI 协作才最有效？**

抄没抄的我不评价，但背后有个更值得想的问题——我们和 AI 一起写代码这件事，到底进化到哪了？

---

## 三个层次

我从 2023 年底开始认真用 AI 辅助编程，到现在两年多，经历了三个阶段。每个阶段感觉完全不一样。

**Chat 模式**——你写提示，AI 给代码，你复制粘贴回去。最早期的玩法，ChatGPT 和 Claude 的网页端就是干这个的。

**Agent 模式**——AI 能读你的项目文件、跑命令、自动修复错误。你只用说"帮我加一个搜索功能"，它自己读代码结构、找相关文件、改代码、跑测试。Claude Code、Cursor Agent、Codex CLI 都是这个路数。

**Multi-Agent 模式**——多个 AI 角色分工协作：一个写代码、一个查文档、一个跑测试、一个做安全审查。理论上各司其职，实际上……后文再聊。

这三个层次不是替代关系，是**叠层关系**。Chat 模式在任何场景都能用，Agent 模式在更复杂的任务上更有用，Multi-Agent 在特定场景下才有意义。

![AI编程工作流进化路径](/images/ai-coding-evolution.svg)

💡 **换个角度理解**：Chat 模式像计算器——按一个键得一个结果，每一步都要手动操作。Agent 模式像写脚本——把逻辑描述清楚，它自动跑完。Multi-Agent 像有个小团队——有产品经理、开发、测试，各做各的事。

## 对比：一图看懂三种模式的差异

| 维度 | Chat 模式 | Agent 模式 | Multi-Agent |
|------|-----------|------------|-------------|
| 交互方式 | 对话 + 手搓 | 指令 + 自动执行 | 任务分解 + 分工协作 |
| 代表工具 | ChatGPT、Claude Web | Claude Code、Cursor Agent、Codex CLI | AutoGPT、CrewAI、自定义pipeline |
| 典型任务 | 写段代码、改算法 | 加功能、重构模块 | 调研报告、全栈项目 |
| 控制力 | ★★★★★ | ★★★☆☆ | ★★☆☆☆ |
| 学习成本 | 零 | 低 | 高 |
| 犯错成本 | 低 | 中 | 高 |

Chat 模式的优点是控制力强——每行代码都过你的手。缺点是慢，零散的复制粘贴打断思路，而且你很难做大事——改个接口名称要粘几十次。

![三种模式对比](/images/ai-modes-compare.svg)

Agent 模式把"执行"这件事外包了，你负责"决策"——描述需求、审核结果。效率提升非常明显，但对你的判断力要求更高。

Multi-Agent 目前还在早期阶段，我后面专门说。

## Agent 模式的真实体验

用一个真实案例说话。

之前要部署一个 Telegram Bot，功能是定时抓取某个网站的价格变化，推送到群里。

**过去用 Chat 模式的流程**：
查文档 → 找到示例代码 → 复制到本地 → 改配置 → 试运行 → 报错 → 回头看文档 → 改 → 再试 → 又报错 → 继续查……一个下午没了。累的不是写代码，是来回切换上下文。

**用 Agent 模式的流程**：
"在这个项目目录里帮我部署一个 Telegram 价格监控 Bot，抓取 xx 网站的价格，每小时检查一次，有变动推送到群里。"

就这一句话。

![Agent工作流](/images/agent-workflow.svg)

剩下的它自己读代码、搜文档、装依赖、改配置、试跑、报错了自己修、修完再跑。前后不到 10 分钟。

跑了几次之后的感觉是：**Agent 模式对"你知道该怎么做但懒得动手"的任务，效率提升非常明显**——它把执行层全包了。但对"你也不知道该怎么做"的任务，它帮不上太多忙。因为需要你来判断方向，而方向判断这件事，目前 AI 做得还很差。

💡 **一个实用技巧**：我现在的标准工作流是先自己想想架构，写个 Todo list，然后交给 Agent 逐条执行。中间随时看进度、纠正方向。这样既保持了控制力，又用上了 Agent 的执行效率。

## 硬约束：Agent 不是万能的

用了一年多，总结几个真实痛点。

**第一，Token 消耗很大。**

同样改一个函数，Chat 模式花几百 token。Agent 模式要读整个文件结构、了解上下文、试跑几次——一次对话下来几万 token 是常事。用 Claude Code 做一天活，账单可能比你的午饭还贵。**这不是效率问题，是成本问题。**

**第二，上下文长度是硬天花板。**

模型吹得再厉害，实际超过 40-50k token 后，Agent 就开始"失忆"了。它会忘掉你最早说的需求，或者把两个不相关的文件混在一起改。你花 10 分钟跟它解释上下文，最后发现生成的东西不对——那种感觉真的很憋屈。

**第三，容易陷入修复循环。**

Agent 发现 bug → 尝试修 → 引入新 bug → 再修 → 再引入……几分钟后你不知道它在改什么了。我遇到过最夸张的一次，它在同一个函数上反复改了 7 次，每次都在修上一个改动引入的新问题。**这时候最有效的操作是 Ctrl+C 重来。**

**第四，它不理解"还没写"的部分。**

Agent 只能看到当前已有的代码。对于你要加但还没写的东西、你脑子里但没落到文件上的设计，它一概不知。所以它能做好"修改"和"扩展"，但做不好"从零构建"。

## 适用场景清单

写这篇文章之前我特意复盘了一下过去半年用 AI 写代码的场景，画了一条线。

**适合用 Agent 模式：**
- ✅ 重构已知架构的代码（换个框架、改个接口风格），这个场景下它比人快 3-5 倍
- ✅ 写单元测试——这可能是我用过最强的场景。给它一个函数，它自动分析边界条件、写测试用例、mock 依赖、跑覆盖率
- ✅ 批量改文件（重命名、加日志、改注释、统一风格）
- ✅ 读代码写文档——Agent 在理解已有代码然后生成文档这件事上，比大多数人做得都好

**不建议用 Agent 模式：**
- ❌ 设计新系统的架构。概念设计需要全局视角，Agent 目前做不到
- ❌ 需要深度业务理解的任务。如果代码的成败取决于你对某个领域知识的理解，别交给 Agent
- ❌ 安全敏感的代码——你至少要做一次人工审查。我见过 Agent 写出硬编码的 API 密钥和 SQL 注入漏洞
- ❌ Token 预算紧张的时候。不是说省钱第一，而是你心里要有数

## 聊聊 Multi-Agent：理想很丰满

Multi-Agent 是 2025 年下半年开始热起来的概念。逻辑很顺——像软件开发团队一样，让不同角色的 Agent 各司其职。

我试过 CrewAI 和自定义的 Multi-Agent pipeline。

**做得好的地方**：研究型任务确实有优势。比如"调研一下这份技术方案的安全性"，一个 Agent 读代码，一个查 CVE 数据库，一个写报告。三个一起做，比一个快很多。

**做得不好的地方**：协调成本非常高。Agent 之间会互相改对方的代码，改出冲突。两个 Agent 对同一个问题的理解可能不一致，导致做出来的东西前后矛盾。我遇到过最无语的情况——一个 Agent 加了新功能，另一个觉得它破坏了代码风格，又给改回去了，然后卡死了。

目前我的判断是：Multi-Agent 更适合做**信息处理类任务**（调研、分析、报告生成），而不是直接写生产代码。要让它可靠地写生产代码，需要更完善的冲突解决机制和任务编排能力——这些东西目前还在探索阶段。

## 聊聊我的感受

用 AI 写代码两年多，最深的体会不是技术层面的。

**工具的价值不取决于它的能力，取决于你用它做什么。**

这话听起来像鸡汤，但体验下来是真的。Chat 模式在很多人手里一样能写出好代码，Agent 模式在有些人手里也只会生成垃圾。关键不在于你用的工具多酷，而在于你有多清楚自己要什么。

我见过一些人，GPT 出来后就说"程序员要失业了"，然后每天用 AI 生成几百行自己都看不懂的代码。也见过一些人，只用最基本的 Chat 模式，但每个提示都经过仔细思考，生成的代码比大多数人手写的都好。

AI 编程工具解决的是"怎么写"的问题，但"写什么"和"为什么写"这两个问题，仍然完全取决于你。

如果你刚开始接触 AI 编程，我的建议是：**先别急着升级工具。** 先用 Chat 模式把每个提示写清楚，学会怎么拆解需求、怎么审核 AI 的输出。等这些基本功到位了，再升级到 Agent 模式——这时候你才知道怎么用好它。

未来呢？我觉得 Agent 模式会成为标配。就像今天我们不会手动编译代码一样，未来可能也不会逐文件手动写代码——描述需求、审核结果会成为主要工作方式。

但在这个未来到来之前，**把基本功练好永远不亏**。因为越是依赖 AI，你越需要知道它做得好不好。

以上是个人经验，仅供参考。有不同的感受欢迎交流。
